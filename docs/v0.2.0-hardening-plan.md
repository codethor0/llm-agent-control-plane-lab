# v0.2.0 Security Hardening Plan

Planned maturity increment for **llm-agent-control-plane-lab**. This is a research and engineering roadmap, not a commitment date.

**Prerequisite baseline:** v0.1.2 (84 tests, E2E-verified, Mermaid architecture docs, SECURITY-CONTROLS matrix).

**Scope rule:** Every new control ships with tests. If it is not tested, it does not exist ([PROJECT_DOCTRINE.md](../PROJECT_DOCTRINE.md)).

## Release theme

**Security Hardening v0.2.0** — depth over aesthetics. No change to core thesis: the model can ask; the broker decides.

## Priority order

| Priority | Work package | Rationale |
|----------|--------------|-----------|
| P0 | Policy integrity | Broker is useless if policy file is wrong |
| P1 | Signed or hashed provenance | Closes largest trust gap in retrieval path |
| P2 | Tool-output injection tests | Doctrine claims distrust; needs proof |
| P3 | Approval token model | Realistic human-in-the-loop semantics |
| P4 | Output filter layers | Incremental, test-per-rule |
| P5 | Hypothesis fuzz tests | Find validator edge cases |
| P6 | Supply-chain tooling | Dependabot, CodeQL, gitleaks, Trivy, SBOM |
| P7 | Production hardening doc | Safe deployment boundaries for demo API |
| P8 | LLM adapter interface | Integration path without live APIs by default |

## P0: Policy integrity — Done

### Delivered

- `src/agent_control_plane/policy_integrity.py` — schema, invariants, canonical SHA-256
- `scripts/validate_policy.py` — validate + compare to `policies/default.sha256` (`--write-hash` for intentional updates)
- `make policy-integrity` wired into `make validate`
- CI step: Policy integrity
- `tests/test_policy_integrity.py`

### Acceptance criteria (met)

- Invalid policy file fails CI and `validate_policy.py`
- Unknown top-level keys and missing required fields rejected
- Golden hash mismatch fails validation
- Existing policy engine behavior unchanged for valid `default.yaml`

### Honest limitations

- SHA-256 detects drift only; not signed attestation
- `policy_hash` audit event on pipeline start not implemented (optional future)

### Out of scope (unchanged)

- HSM-backed policy signing

## P1: Provenance attestation (lab-safe)

### Deliverables

- `ProvenanceAttestation` model: `content_hash`, `retrieval_event_id`, `signer_id`, `signature` (HMAC with lab key from env example only)
- Verification in `provenance.py` before policy treats source as `internal_reviewed`
- Docs update in [provenance.md](provenance.md) — still not production PKI

### Acceptance criteria

- Forged signature denied
- Hash mismatch denied
- Valid lab HMAC allows only tools already allowed by policy
- No real external services

### Tests

- `test_provenance_signature_required_for_internal_reviewed`
- `test_provenance_hash_mismatch_denied`

## P2: Tool-output injection

### Deliverables

- New scenarios in `agent_core.py` or pipeline fixtures: tool result text with instruction-like content
- Multi-turn simulation stub (in-memory only): turn 1 tool output feeds turn 2 prompt assembly
- Tests prove broker/filter still block unauthorized follow-up

### Acceptance criteria

- No test weakens existing injection_send_email coverage
- Documented in [defensive-controls.md](defensive-controls.md)

## P3: Approval token model

### Deliverables

- `ApprovalToken` dataclass: `approval_id`, `approver_id`, `issued_at`, `expires_at`, `action_hash`, `target_hash`, `used`
- Broker checks token matches current tool call fingerprint
- Audit events: `approval_granted`, `approval_denied`, `approval_reused`

### Acceptance criteria

- Boolean `human_approval` remains supported for backward compatibility OR migrated with deprecation note in release notes
- All approval gate tests extended, not weakened

## P4: Output filter layers

### Deliverables (incremental)

1. High-entropy string heuristic (with false-positive tests)
2. Tenant marker in output when request tenant does not match (simulated)
3. Document destination-aware rules as future hook

### Acceptance criteria

- Each rule has pass and fail tests
- `tests/test_output_filter.py` count increases; no skips

## P5: Hypothesis fuzzing

### Deliverables

- `tests/test_fuzz_schemas.py` — valid tool args round-trip
- `tests/test_fuzz_targets.py` — email and tenant target validation
- Optional: fuzz output filter strings (bounded examples)

### Dependency

- Add `hypothesis` to dev extras only if approved for repo

## P6: Supply-chain

### Deliverables

- `.github/dependabot.yml` — pip and GitHub Actions
- `.github/workflows/codeql.yml` — Python analysis
- gitleaks or equivalent in CI (allowlist false positives)
- Trivy scan on Docker image in CI
- `scripts/generate_sbom.sh` — CycloneDX via pip-audit or similar
- `docs/supply-chain.md` — pinned actions policy, signed tag guidance

### Acceptance criteria

- CI green with new jobs
- No secrets in repo; scanner passes

## P7: Production hardening documentation

### Deliverables

- [production-hardening.md](production-hardening.md) (new)
- Link from README Safe use section

### Topics

- Authentication and authorization required for any network exposure
- Rate limiting and request size limits
- CORS and TLS termination outside app
- Network isolation; non-root container; read-only filesystem
- Explicit warning: demo API is not production-ready

### No runtime auth in v0.2.0 unless separately approved

## P8: LLM adapter interface

### Deliverables

- `src/agent_control_plane/llm_client.py` — protocol only
- `LocalFakeLLMClient` — wraps current `agent_core`
- `ExternalLLMClient` — raises `NotImplementedError`
- [docs/llm-adapter.md](llm-adapter.md) — safe integration guide

Expected test count increases; document new total in release notes.

## Release versioning

- Tag: `v0.2.0` when P0–P3 minimum complete (policy integrity, provenance attestation lab mode, tool-output tests, approval tokens)
- P4–P8 may ship in `v0.2.x` patches if split for reviewability

## Honest limitations after v0.2.0 (projected)

- Still a lab; not certified for arbitrary production deployment
- HMAC provenance is educational, not enterprise PKI
- Simulated tools only unless explicitly extended with new tests
- Live LLM integration remains opt-in and documented
