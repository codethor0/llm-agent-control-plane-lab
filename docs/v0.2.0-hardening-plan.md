# v0.2.0 Security Hardening Plan

Planned maturity increment for **llm-agent-control-plane**. This is a research and engineering roadmap, not a commitment date.

**Prerequisite baseline:** v0.1.2 (84 tests, E2E-verified, Mermaid architecture docs, SECURITY-CONTROLS matrix).

**Scope rule:** Every new control ships with tests. If it is not tested, it does not exist ([PROJECT_DOCTRINE.md](../PROJECT_DOCTRINE.md)).

## Release theme

**Security Hardening v0.2.0** â€” depth over aesthetics. No change to core thesis: the model can ask; the broker decides.

## Priority order

| Priority | Work package | Rationale |
|----------|--------------|-----------|
| P0 | Policy integrity | Broker is useless if policy file is wrong |
| P1 | Signed or hashed provenance | Closes largest trust gap in retrieval path |
| P2 | Tool-output injection tests | Doctrine claims distrust; needs proof |
| P3 | Approval token model | Realistic human-in-the-loop semantics |
| P4 | Output filter layers | Incremental, test-per-rule |
| P5 | Hypothesis fuzz tests | Find validator edge cases |
| P6 | Supply-chain tooling | Done |
| P7 | Production deployment hardening | Done |
| P8 | LLM adapter interface | Integration path without live APIs by default |

## P0: Policy integrity â€” Done

### Delivered

- `src/agent_control_plane/policy_integrity.py` â€” schema, invariants, canonical SHA-256
- `scripts/validate_policy.py` â€” validate + compare to `policies/default.sha256` (`--write-hash` for intentional updates)
- `make policy-integrity` wired into `make validate`
- CI step: Policy integrity
- `tests/test_policy_integrity.py`

### Acceptance criteria (met)

- Invalid policy file fails CI and `validate_policy.py`
- Unknown top-level keys and missing required fields rejected
- Golden hash mismatch fails validation
- Existing policy engine behavior unchanged for valid `default.yaml`

### Honest limitations

- SHA-256 detects drift only; not signed attestation
- `policy_hash` audit event on pipeline start not implemented (optional future)

### Out of scope (unchanged)

- HSM-backed policy signing

## P1: Provenance attestation (lab-safe) â€” Done

### Delivered

- `provenance_integrity.py` â€” canonical JSON, HMAC-SHA256 sign/verify, `compute_content_hash`
- Optional `tenant_id`, `chunk_id`, `content_hash`, `signature` on `Provenance`
- `require_provenance_signature` on broker and pipeline (default off)
- `tests/test_provenance_integrity.py` (17 tests)
- [provenance.md](provenance.md) updated

### Acceptance criteria (met)

- Tampered tenant, trust, source, or content_hash fails verification
- Strict mode fails closed on missing/invalid signature
- Valid signature does not bypass broker policy
- Signed sources still follow declarative authorization rules

### Honest limitations

- `LAB_DEMO_HMAC_KEY` is a fake lab constant, not production key management
- Default `run_protected` does not require signatures

## P2: Tool-output injection â€” Done

### Delivered

- `ToolOutputSegment` on `AgentRequest`; untrusted section in `prompt.py`
- `ProvenanceSource.TOOL_OUTPUT` cannot authorize tools (`provenance.py`)
- Scenarios in `agent_core.py` for send_email, shell, cross-tenant, and secret echo paths
- `tests/test_tool_output_injection.py` (13 tests)

### Acceptance criteria (met)

- Tool output cannot authorize external effects or override policy via provenance
- Output filter blocks fake secrets echoed from tool-output context
- Protected pipeline scenarios blocked at broker or output filter
- Existing `injection_send_email` tests unchanged

### Honest limitations

- Simulated paths only; not all production tool integrations covered
- Multi-turn coverage is in-test simulation, not a live second pipeline turn in production code

## P3: Approval token model â€” Done

### Delivered

- `approval_tokens.py` â€” `ApprovalToken`, fingerprints, create/verify, one-time use registry
- `AgentRequest.approval_token` optional field; `require_approval_token` on broker and pipeline
- Audit fields: `approval_id`, `approver_id`, `approval_decision`, `approval_reason`, `approval_token_valid`, `approval_token_failure_reason`
- `tests/test_approval_tokens.py` (21 tests)

### Acceptance criteria (met)

- Token bound to tool, target, tenant, user, risk, context IDs, provenance fingerprint, action fingerprint
- Expired, reused, and mismatched tokens denied
- Token does not bypass policy, provenance signature, or disabled tools
- Legacy `human_approval` boolean still works when no token is required

### Honest limitations

- No production IdP or approval UI; in-process one-time registry only

## P4: Output filter layers

**Status:** Done on `feature/p4-output-filter-layers` (pending merge to `main`).

### Deliverables (incremental)

1. High-entropy string heuristic (with false-positive tests)
2. Tenant marker in output when request tenant does not match (simulated)
3. Destination-aware rules (`internal_display`, `audit_log`, `external_email`, `external_export`, `webhook`)
4. Structured `OutputFinding` model with redacted samples
5. Source sensitivity propagation from retrieved chunks and tool output
6. Allowlisted response schema checks (strict mode)
7. Audit metadata for output filter decisions (redacted findings only)

### Acceptance criteria

- Each rule has pass and fail tests
- `tests/test_output_filter.py` count increases; no skips (23 tests as of P4 branch)
